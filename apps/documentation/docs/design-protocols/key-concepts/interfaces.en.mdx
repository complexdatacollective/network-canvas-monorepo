<Definition>
  Data collection instruments designed for a specific task. You configure
  Interfaces in Architect and add them to the interview protocol, where they
  become stages.
</Definition>

## Details

Interviews in Network Canvas are comprised of one or more stages, which can be thought of as the individual screens that a participant sees. Each stage is an _instance_ of an Interface that has been configured to collect specific [variables](/en/design-protocols/key-concepts/variables) or create nodes and edges. Interfaces are therefore the most fundamental building blocks of a Network Canvas interview.

Each Interface is designed to accomplish a single specific task in the network interview. Some tasks, such as edge generation, can be accomplished with different Interfaces. You choose which Interface to use when designing your interview in Architect based on your research needs.

## Types of interfaces

Interfaces can be broadly categorized as: name generators, sociograms, name interpreters, or utilities.

### Name Generator interfaces

Name Generators allow participants to create alters during the course of an interview, thereby describing one of the two main entities in your study's networks. There are five Name Generator Interfaces available, each designed for specific scenarios:

- [Name Generator (using forms)](/en/design-protocols/interface-documentation/name-generator-using-forms) - Collect detailed information about each alter as they are named
- [Name Generator (using quick add)](/en/design-protocols/interface-documentation/name-generator-using-quick-add) - Quickly name multiple alters before collecting attributes
- [Small Roster Name Generator](/en/design-protocols/interface-documentation/small-roster-name-generator) - Select alters from a predefined list (best for rosters under 100)
- [Large Roster Name Generator](/en/design-protocols/interface-documentation/large-roster-name-generator) - Select alters from large rosters with search functionality
- [Family Tree Census](/en/design-protocols/interface-documentation/family-tree-census) - Build a family pedigree by creating family members and the relationships between them, then collect attributes for each. (This interface is labelled **Family Pedigree** in Architect; it also generates edges and interprets alters.)

![Available Name Generator Interfaces in Architect](/assets/img/key-concepts/interfaces/name-generators.png)

### Sociogram interfaces

Sociograms are the visual mapping component of a network interview. On the Sociogram, participants draw ties between alters to signify the relationships between individuals within their personal network, describing the other main entity in your study's networks.

There are two Sociogram Interfaces in Network Canvas:

- [Sociogram](/en/design-protocols/interface-documentation/sociogram) - Interactive Interface for collecting edge data by having participants draw connections between alters
- [Narrative](/en/design-protocols/interface-documentation/narrative) - Display-only Interface for showing the participant's network without collecting data

![Available Sociogram Interfaces in Architect](/assets/img/key-concepts/interfaces/sociograms.png)

### Name Interpreter interfaces

Once a protocol contains entities, Name Interpreter Interfaces can be used to collect additional information and build upon existing alters or ties. These Interfaces include:

- [Ordinal Bin](/en/design-protocols/interface-documentation/ordinal-bin) - Sort alters into ordered categories using a visual bin interface
- [Categorical Bin](/en/design-protocols/interface-documentation/categorical-bin) - Sort alters into unordered categories using a visual bin interface
- [Per Alter Form](/en/design-protocols/interface-documentation/per-alter-form) - Collect detailed attribute data about each alter using [forms](/en/design-protocols/key-concepts/forms)
- [Per Alter Edge Form](/en/design-protocols/interface-documentation/per-alter-edge-form) - Collect attribute data about edges using forms
- [Ego Form](/en/design-protocols/interface-documentation/ego-form) - Collect data about the participant themselves
- [Dyad Census](/en/design-protocols/interface-documentation/dyad-census) - Systematically ask about relationships between all pairs of alters
- [One-To-Many Dyad Census](/en/design-protocols/interface-documentation/one-to-many-dyad-census) - A Dyad Census variation that links one alter to multiple others at a time
- [Geospatial](/en/design-protocols/interface-documentation/geospatial) - Capture geospatial data by having participants select areas on a map (requires an internet connection and a Mapbox account)

![Available Name Interpreter Interfaces in Architect](/assets/img/key-concepts/interfaces/name-interpreters.png)

### Utility interfaces

The [Information Interface](/en/design-protocols/interface-documentation/information) is a general purpose stage that can be used to present information to participants using a variety of text and media [resources](/en/design-protocols/key-concepts/resources). This is helpful in explaining stage functionalities and increasing participant comprehension.

The [Anonymisation Interface](/en/design-protocols/interface-documentation/anonymisation) is an experimental stage that lets participants encrypt sensitive or identifiable attributes on-device, so that they cannot be read by the researcher. It is not enabled by default; you turn it on via the **Encrypted Variables** experimental feature in Architect.

![Available Utility Interfaces in Architect](/assets/img/key-concepts/interfaces/utilities.png)
