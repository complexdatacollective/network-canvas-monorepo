# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in `apps/interviewer`.

See [`SPEC.md`](./SPEC.md) for the product specification and [`README.md`](./README.md) for architecture diagrams, data flows, invariants, and tradeoffs.

## Project Overview

Network Canvas Interviewer is a single-user, offline-first research-data-collection app that hosts the `@codaco/interview` engine. It ships as one thing: an installable **web PWA** (Vite + `vite-plugin-pwa`), deployed to Netlify. There is no Electron desktop build and no Capacitor tablet build — both were retired; see `RELEASING.md`'s "What used to be here" if you're looking for the old release process.

Storage is Dexie 4 / IndexedDB, encrypted at rest by an in-app Web Crypto vault (see Vault & Auth below) — there is no native SQLCipher layer and no main/renderer process split.

## Development Commands

All commands run from this directory unless noted. The monorepo-wide `pnpm lint` / `pnpm typecheck` from the repo root also cover this app.

```bash
pnpm dev              # Vite dev server (workspace deps consumed as raw src, no build step)
pnpm build            # production build + PWA integrity check (node scripts/build.mjs)
pnpm preview           # preview the production build locally
pnpm typecheck         # tsc --build --noEmit
pnpm test              # vitest run --project=unit
pnpm test:watch        # vitest --project=unit, watch mode
pnpm test:storybook    # vitest run --project=storybook
pnpm storybook         # Storybook dev server on :6006
```

`build` is the command CI deploys with. It runs Vite and then fails if the service worker, manifest, or icons are missing from `dist/`, or if a critical JS chunk (interview engine, entry point) got silently dropped from the workbox precache manifest. Treat an `assert-pwa-build.mjs` failure as a real bug, not noise.

## Why this structure

- **`src/lib/db/api.ts` facade**: keeps routes/components ignorant of the Dexie repo internals and the encrypt/decrypt step that `recordCrypto.ts` inserts on every read/write.
- **Field-level encryption, not whole-row**: `recordCrypto.ts` encrypts only session `network`/`stageMetadata`, protocol `protocol`/`codebook`, and asset `data`. Index fields (`caseId`, `protocolName`, timestamps, status) stay plaintext so Dexie can query/sort/filter them directly without decrypting every row.
- **In-memory-only session DEK**: `sessionKey.ts` holds the unlocked data-encryption key in a module-scope variable, nothing more durable. A page reload drops it — the app re-locks on every reload by design (see Vault & Auth).
- **Effect-TS export pipeline**: `Layer.mergeAll` cleanly assembles the renderer-specific repositories and Blob sink without leaking implementation details into the shared `@codaco/network-exporters` package.
- **`fresco-ui` first**: every Button/Surface/Form/Dialog/Toast or typography token should reach for `@codaco/fresco-ui` first; fall back to `@base-ui/react` + `motion` only for primitives fresco-ui doesn't ship (fanned deck, translucent-blur pills, animated blob background).

## Vault & Auth

`src/lib/vault/` is a Web Crypto (`crypto.subtle`) vault, not a native keychain. One `VaultRecord` (in `localStorage`, versioned — currently v4) describes exactly one enrolled mode:

- **`pin`** — exactly 8 digits, PBKDF2-HMAC-SHA256 (600k iterations, 32-byte salt) derives a KEK that wraps (AES-KW) a random AES-GCM DEK.
- **`passphrase`** — free-form, ≥ 12 characters with ≥ 3 character classes, same PBKDF2 envelope as PIN.
- **`biometric`** — WebAuthn platform authenticator with the PRF extension. The PRF output (via HKDF) derives a KEK that wraps the DEK. Enrolment **requires** a recovery passphrase in the same step: the same DEK is dual-wrapped, once under the PRF-derived KEK and once under a PBKDF2 KEK from the recovery passphrase — losing the authenticator doesn't mean losing the data. Enrolment fails fast (and drops the orphaned passkey) if the authenticator doesn't report `prf.enabled` at creation. Known platform limitation: Chromium on macOS can't reach the iCloud Keychain authenticator from installed-PWA windows (crbug.com/364926914), so biometric is not offered there and unlock/step-up default to the recovery passphrase — detection lives in `src/lib/pwa/passkeyWindowLimitation.ts`.
- **`none`** — no wrapped DEK exists at all; reads/writes pass through unencrypted. `recordCrypto.ts` fails closed if a secured mode's vault is locked (no plaintext fallback for `pin`/`passphrase`/`biometric`).

**The renderer holds the DEK.** There is no main/renderer split and nothing hides the key across an IPC boundary — the unlocked DEK lives in `sessionKey.ts`'s module-scope variable for the lifetime of the tab. A reload, explicit lock, or the user-configured idle timeout all clear it and re-lock. Locking is governed solely by that idle timeout — there is no separate lock-on-blur; time spent with the tab hidden counts toward the idle window, which `idle.ts` reconciles against wall-clock time on return (background tabs throttle/freeze timers). Another tab revoking/re-enrolling the vault force-locks this tab too (a `storage` event listener), so a stale DEK can never encrypt rows the new vault can't decrypt.

Re-enrolling `pin`/`passphrase` (`reEnrolPin`/`reEnrolPassphrase` in `vault.ts`) rewraps the _same_ DEK under a new KEK — atomic (old record stays valid until the new wrap succeeds), and the caller's already-unlocked session DEK doesn't change. `biometric` has no re-enrol path; switching modes means `revoke()` (destructive — wipes the Dexie database) first.

Step-up auth (re-authenticate before a sensitive action, without relocking the whole app) is `StepUpAuthProvider.requireFreshUnlock()` → `StepUpAuthDialog`, using the `verify*` family (never `setSessionDek`). Three persisted `StoredSettings` flags gate interview boundaries via the shared `SecurityBehaviorControls` (used by both the setup wizard's `Step4Behavior` and `SettingsDialog`): `requireUnlockOnEnter` (default `true`), `requireUnlockOnExit` (default `false`), `requireUnlockOnExport` (default `false`).

## PWA & offline

- **`vite-plugin-pwa`**, `strategies: 'generateSW'`, `registerType: 'prompt'` — the service worker never force-applies. `AppUpdateProvider` polls hourly; a pending update is applied automatically on a fresh load when no interview is in progress, and otherwise surfaced on the version indicator (bottom-right, in `StatusRow`) as an "update available" control with the release changelog — the prompt/auto-apply is withheld while the location is inside `/interview/*`, so an update can't interrupt a session.
- **File handling**: the installed app registers as a `.netcanvas` opener (manifest `file_handlers` + `launch_handler: focus-existing`; Chromium desktop only — Safari has no equivalent). Launched files are captured pre-React by `src/lib/pwa/fileLaunchQueue.ts` and imported by Home once the vault is unlocked.
- **Install banner**: `InstallBanner` (mounted on Home) urges installing before collecting data and maps browser-storage danger to a shared intent: WebKit's proactive cleanup is high risk, Firefox's storage-pressure eviction is medium risk, and Chromium's rare automatic eviction is low risk. Installation protects against routine cleanup. One-tap install uses `src/lib/pwa/installPrompt.ts` (`beforeinstallprompt` where supported; Apple platforms get Share-menu instructions). Dismissal lasts one session.
- **Storage durability**: `src/lib/storage.ts` requests persistent (non-evictable) storage and surfaces usage/quota in Settings and `StatusRow`.
- **Offline UX**: `useOnline` (re-exported through `OnlineStatusProvider` from `@codaco/interview`) is the single source of online/offline state app-wide. `protocolRequiresInternet.ts` flags protocols with a Geospatial stage (online map tiles); `NewSessionForm` warns before starting one offline. The Geospatial stage's own offline error boundary and persistent offline indicator live in the `@codaco/interview` package, not here.
- **Netlify**: `public/_headers` (no-cache on `sw.js`/`index.html`/manifest/icons so a release always revalidates; long-cache immutable on hashed `assets/*`) and `public/_redirects` (SPA fallback to `index.html`). See `RELEASING.md` for the deploy pipeline.

## Conventions

- **No `any`, no `as` assertions to bypass type checking.** Fix the underlying type instead.
- **No barrel files** (`index.ts` re-export aggregators). Import from the original source.
- **No re-exports for convenience.** All references to a function/variable should import from the original source.
- **oxlint + oxfmt** lint and format: 2-space indentation, single quotes. Pre-commit hooks format staged files. Run `pnpm lint:fix` from the repo root before committing.
- **Tests** are co-located in `__tests__/` directories with `.test.ts` / `.test.tsx` extensions, using Vitest.
- **Four auth modes, never combined.** A vault is exactly one of `pin` / `passphrase` / `biometric` / `none` — see Vault & Auth above.
- **Single-user invariant.** `installationId` identifies the device, not a user. No code path may introduce a user identifier.
- **No cloud sync, no server component.** Everything is on-device; the only network calls are protocol-embedded map tiles (Geospatial) and the opt-in PostHog relay.
