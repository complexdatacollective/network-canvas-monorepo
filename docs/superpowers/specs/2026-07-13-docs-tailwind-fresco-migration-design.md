# Documentation app — shared-theme + fresco-ui migration — design

**Date:** 2026-07-13
**Status:** Approved design (autonomous implementation authorised by maintainer)
**Scope:** `apps/documentation` (full adoption of the shared design system) **plus** authoring a canonical default **dark** theme in `tooling/tailwind` (the shared package). Single PR.

## Context and goal

`apps/documentation` is a Next.js 16 App Router site (React 19, static export, next-intl), styled with Tailwind **v4** via `@tailwindcss/postcss`. Unlike `apps/architect` and `apps/interviewer`, it consumes **none** of the shared design system: it carries a **local fork** of the fresco theme — a `@config '../tailwind.config.ts'` bridge to `tailwind/fresco.ts` + `tailwind/base.ts`, plus ~250 lines of inline HSL palette + semantic-slot blocks in `styles/globals.css` — and its own Radix/shadcn/cva primitives under `components/ui/`. It depends on neither `@codaco/fresco-ui` nor `@codaco/tailwind-config`.

**Goal:** make the documentation app a first-class consumer of the shared design system. Delete the theme fork and the bespoke primitives; consume `@codaco/tailwind-config/fresco.css` + `@codaco/fresco-ui`; reuse fresco-ui components everywhere except genuinely docs-unique machinery. **Full adoption**: Inclusive Sans + Nunito fonts, the shared oklch palette, and the shared dark mode — the site ends up looking like the rest of the system (architect/interviewer), not like its former self.

Because a working default dark theme does not exist anywhere in the system today (architect is light-only; interviewer uses the interview theme; the shared `[data-theme='dark']` block is broken and unused), the migration also **authors** that dark theme in the shared package, with docs as its first consumer.

### Reference precedent

`docs/superpowers/specs/2026-07-08-architect-shared-theme-migration-design.md` established the foundation pattern (the `@import` trio, `@theme inline` re-resolution rules, the shadcn→fresco token renames R1, the forbidden-pattern grep gate R7, the foundation-spike-gates-everything shape). This migration reuses that vocabulary. Differences: docs is **full adoption** (no permanent brand-override layer — architect kept its identity; docs does not), docs **does** change `tooling/tailwind` (the dark theme — architect explicitly excluded it), and docs **replaces** primitives with fresco-ui components (architect deferred that).

## Decisions

Resolved with the maintainer 2026-07-13:

1. **Visual intent — full adoption.** Adopt the shared theme wholesale: Inclusive Sans + Nunito fonts, oklch palette, shared dark mode. No local theme fork, no brand-override layer. Visible change to fonts and dark mode is accepted and intended (design-system consistency).
2. **Dark mode — author canonical dark in the shared package.** Rewrite the broken `[data-theme='dark']` block in `tooling/tailwind/fresco/themes/default.css` into a real, oklch-derived dark variant, built the way the light theme is (from `--brand`/`--base-hue`). The docs site's current dark palette is the visual quality bar. This benefits the whole system; docs is its first consumer.
3. **Delivery — single PR.** One PR carries both the shared dark-theme change and the docs migration, with **two separate changeset files**: `@codaco/tailwind-config` **minor** (library) and `@codaco/documentation` **minor** (gated product — in the changeset `ignore` list but releases with normal semver via the "Release apps & documentation" PR). The lanes must not be combined in one file (`check:changesets` rejects mixed changesets), but two single-lane files in one PR is allowed. Internal ordering: (1) shared dark theme → (2) docs foundation swap → (3) component migration.
4. **Component reuse — replace where a genuine equivalent exists, keep docs-unique machinery.** Local generic primitives (`components/ui/*`) are replaced by fresco-ui at call sites; docs-flavoured callouts keep their MDX-author-facing API but are rebuilt on fresco-ui primitives; genuinely unique machinery (the MDX renderer, sidebar/TOC/search, workflow/app-switch nav, syntax-highlighted code blocks, `@codaco/art` blobs) stays local and is only re-themed to fresco tokens.

## Architecture

### Stylesheet stack — `apps/documentation/styles/globals.css` (rewritten)

```css
@import '@codaco/tailwind-config/fresco.css'; /* bundles Tailwind + palette + @theme + default/interview themes + TS plugins */
@import '@codaco/fresco-ui/styles.css';       /* @source glue so Tailwind emits classes fresco-ui's dist references */

/* Docs-unique: long-form MDX prose styling (shared theme does not cover this). */
@plugin '@tailwindcss/typography';
/* …docs prose @theme / .prose token bindings to fresco tokens (see Component migration) */
```

Removed from `globals.css`: `@config '../tailwind.config.ts'`, `@import 'tailwindcss'` (forbidden second import — `fresco.css` already pulls Tailwind), the ~250 lines of inline `--x-hue/-saturation/-lightness` palette vars and the `@layer base` semantic-slot blocks, `@import 'tw-animate-css'` (superseded by the shared `motion-spring` plugin + shared keyframes — pending usage check), and the local `@utility focusable` (shared `utilities.css` provides it). Kept: the `tokyo-night-dark` syntax-highlight stylesheet (orthogonal to the app theme).

`postcss.config.mjs` stays (`@tailwindcss/postcss` is how Next runs Tailwind v4). `@tailwindcss/container-queries` plugin is dropped (built into v4).

### Deleted fork

`apps/documentation/tailwind.config.ts`, `apps/documentation/tailwind/base.ts`, `apps/documentation/tailwind/fresco.ts`, `apps/documentation/components.json` (shadcn config, vestigial once `components/ui/*` are retired), and the inline palette/slot blocks in `globals.css`.

### Dependencies

Add to `apps/documentation/package.json`: `@codaco/fresco-ui: workspace:*`, `@codaco/tailwind-config: workspace:*`. Remove `@fontsource-variable/quicksand` and — as their consumers are retired — the now-unused `@radix-ui/*` / `class-variance-authority` / `tailwind-merge` / `clsx` direct deps that only the local primitives used (verified per-dep during the component pass; `knip` is the gate). `@base-ui/react` is already present (fresco-ui peer).

### Fonts / entry wiring

Import `@codaco/tailwind-config/fonts/inclusive-sans.css` and `.../nunito.css` from a JS entry (so `url()`s resolve through the bundler, mirroring `apps/interviewer/src/main.tsx`), replacing the `@fontsource-variable/quicksand` import in `app/[locale]/layout.tsx`. The shared theme already binds `--font-body`/`--font-heading` to `'Inclusive Sans Variable'`/`'Nunito Variable'`; remove the local `fontFamily.sans = Quicksand` mapping (it dies with `tailwind.config.ts`).

### Dark-mode wiring

Switch `next-themes` from `attribute="class"` to `attribute="data-theme"` (values `light`/`dark`) in `app/[locale]/layout.tsx` and `app/not-found.tsx`, since the shared theme keys dark on `[data-theme='dark']`. The `storageKey` stays. The handful of `dark:` utility usages (4 source files: `WorkflowNav.tsx`, `SharedNav/SharedNav.tsx`, `Layout.tsx`, `lib/docs.tsx`) migrate to semantic tokens that auto-swap (`bg-surface-1`, `text-text`, …) rather than relying on a `dark:` variant.

### Canonical dark theme — `tooling/tailwind/fresco/themes/default.css`

Rewrite the `[data-theme='dark']` block. Current block is dead/broken: bare channel triplets (`--background: 19% 0.03 var(--base-hue)`) that don't resolve to valid colors under the current `@theme inline` (`--color-background: var(--background)` — no `oklch()` wrapper), an undefined `--accent-hue`, `--surface-0` that light never declares, and missing most slots.

New block, mirroring the light theme's construction (all values full `oklch()` colors, derived from `--brand`/`--base-hue` where the light theme derives, every semantic slot the light `:root` declares also declared here):

- `--background` low-L (dark), `--text` high-L (light) — inverse of light.
- Surfaces ascend in L on dark (`--surface` darkest → `--surface-3` lightest), opposite of light's descent; each with a light `-contrast`.
- `--primary`, `--accent`, `--secondary`: raise L and trim chroma for legibility on dark; keep hue. Contrasts stay readable.
- States (`--destructive`, `--warning`, `--success`, `--info`) and `--link`: nudge L up for dark backgrounds; keep hue.
- `--input`, `--input-contrast`, `--outline`, `--selected`, `--overlay`, `--surface-popover` re-derived for dark.
- Sequence colors (`--node-*`, `--edge-*`, `--ord-*`, `--cat-*`) inherit from light unless a specific hue fails contrast on dark; adjust individually only where needed.

Exact oklch values are tuned during implementation against the docs site's current dark palette (the quality bar) with side-by-side visual verification. Architect (light-only, no `[data-theme='dark']`) and interviewer (interview theme) are unaffected — verified by build + visual regression check on both.

### Foundation spike (gates everything)

The docs app is the only **Next.js / `@tailwindcss/postcss`** consumer of the shared theme; every reference app uses `@tailwindcss/vite`. `fresco.css` uses `@plugin '../shared/plugins/*.ts'` (jiti-compiled TypeScript) and `@source` scanning. Before any fan-out: wire the two `@import`s + the two new deps, run `pnpm --filter @codaco/documentation build` (`next build`), and settle empirically whether the `.ts` `@plugin`s load and utilities/`@source` classes emit under Next's PostCSS pipeline. If they don't: fall back (a precompiled plugin entry, or a thin CSS shim that `@import`s the pieces without the `.ts` `@plugin` line). This decision gates the whole migration.

## Token remapping (codemod rules)

Docs uses shadcn-style `--foo`/`--foo-foreground` slots exposed as `bg-*`/`text-*`/`border-*` utilities. Fresco uses `--color-*` inline aliases over primitives. Apply these renames to every Tailwind class position (`className`, `cva`/`cx` maps, `@apply`) and CSS `var()` position; only slots actually referenced in docs source are listed. Mirrors architect **R1**.

| Docs slot / utility                                                                        | Fresco token → utility                                                                                       | Rule                                                                                 |
| ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |
| `--foreground` / `text-foreground`, `bg-foreground`, `border-foreground`                   | `--text` → `text-text`, `bg-text`, `border-text`                                                             | **rename** `foreground`→`text`                                                       |
| `*-foreground` suffix (`primary/secondary/accent/destructive/success/input`-`-foreground`) | `*-contrast`                                                                                                 | **suffix rename** `-foreground`→`-contrast`                                          |
| `--border` / `border-border`                                                               | `--outline` → `border-outline`                                                                               | **rename** `border`→`outline` (fresco base already applies `border-outline` to `*`)  |
| `--background` / `bg-background` etc.                                                      | `--background` → same utility                                                                                | value-only; **strip `hsl(var(--background))`→`var(--background)`**                   |
| `--primary/secondary/accent/destructive/success/warning/info` (base)                       | same names                                                                                                   | direct — all exist in `default.css`                                                  |
| `--card` / `bg-card`                                                                       | **none** → `bg-surface-1` (or `<Surface>`)                                                                   | **no equivalent** — map to surface ladder                                            |
| `--popover` / `--popover-foreground`                                                       | **none** → `bg-surface-popover` / `text-surface-popover-contrast`                                            | rename to surface-popover pair                                                       |
| `--muted`, `--muted-foreground` / `bg-muted`, `text-muted-foreground`                      | **none** → `bg-current/5`·`bg-surface-2/3` (fills); `text-current/70` or Paragraph `emphasis="muted"` (text) | **no equivalent — flag**; fresco has no muted slot by design                         |
| `--ring` / `ring-ring`, `ring-offset-background`                                           | **none** → `focusable` utility                                                                               | **no equivalent — flag**; fresco focuses via `focusable` (outline), not a ring token |
| `--panel` (declared, unused as utility)                                                    | **none**                                                                                                     | drop (no `bg-panel` usage in scope)                                                  |
| `rounded-input`                                                                            | `rounded-sm` / `rounded`                                                                                     | custom radius alias → fresco radius scale                                            |
| named colors (`bg-rich-black`, …)                                                          | `--color-rich-black`, …                                                                                      | direct — fresco exposes the full named palette                                       |
| `[--link:var(--accent-foreground)]` and siblings                                           | `[--link:var(--accent-contrast)]`                                                                            | `-foreground`→`-contrast` inside arbitrary props                                     |

**Cross-cutting:** docs writes semantic colors as HSL triplets read via `hsl(var(--x))` / `color-mix(in oklab, hsl(var(--background)) …)`. Fresco tokens are **already resolved colors**, so every `hsl(var(--…))` wrapper around a semantic slot is removed (affects `TipBox`, `KeyConcept`, `WorkInProgress`, `AppCompatibilityTable`, and the `figure` handler in `lib/docs.tsx`).

**`cn` → `cx`:** fresco-ui exports `cx` from `@codaco/fresco-ui/utils/cva` (clsx-join + `twMerge` via an `onComplete` hook) — functionally equal to the local `cn`. Rename `cn(`→`cx(` app-wide so the app and fresco components share one twMerge config; keep `lib/utils.ts` for its link helpers.

- **Phase 0 — Foundation (serial, gates everything):** author the shared dark theme; rewrite `globals.css` + entry wiring + `next-themes`; add deps; delete the fork; run the build spike; capture pre-migration baseline screenshots (light **and** dark) of the checklist routes from the current branch tip. Everything else blocks on a green spike.
- **Phase 1 — Component migration (parallel shards):** disjoint groups of call sites / components — (a) REPLACE primitives (Button, typography, Divider, collapsible/sheet) at call sites; (b) REBUILD callouts (TipBox/KeyConcept/BestPractices/SummaryCard/WorkInProgress) on fresco-ui primitives keeping MDX APIs; (c) KEEP-LOCAL re-theming (renderer, sidebar, TOC, search, nav, code blocks) to fresco tokens; (d) token-rename codemod across remaining source. No file in two shards.
- **Phase 2 — Cleanup:** delete retired `components/ui/*` + fork files; prune dead deps (`knip`).
- **Phase 3 — Verification:** one `pnpm typecheck` + `pnpm lint:fix` + `pnpm knip` + `pnpm --filter @codaco/documentation build` pass; forbidden-pattern grep sweep; visual sweep (light + dark) vs baselines; adversarial diff review.
- **Phase 4 — Fix loop:** findings become fix agents; repeat Phase 3 until two consecutive clean rounds.

## Verification and acceptance criteria

1. `pnpm typecheck`, lint, `pnpm knip`, and `pnpm --filter @codaco/documentation build` pass.
2. Forbidden-pattern greps return zero.
3. **Visual pass** (headless Chromium against `next dev`/preview; non-negotiable), **light and dark**: homepage, a rendered doc page (prose + headings + lists + links + inline code + a code block), a page exercising each callout (`tipbox`, `keyconcept`, good/bad practices, `summarycard`), the sidebar tree + mobile sidebar, the table of contents, DocSearch modal, the app-switch tabs, and the compatibility table. Fonts are Inclusive Sans/Nunito; colors are fresco tokens; dark mode toggles via `data-theme` and looks intentional (matches the quality bar, not the interview theme).
4. Architect and interviewer are unaffected by the shared dark-theme change (build + spot visual check).
5. Search (Algolia), i18n strings (next-intl), and static export (`output: 'export'`) still work.

## Delivery

Single branch (`claude/docs-tailwind-fresco-migration-2beadc`), single PR, **two** changeset files (never mixed in one file — `check:changesets` rejects that): `@codaco/tailwind-config` **minor** (library lane — adds the canonical dark theme) and `@codaco/documentation` **minor** (gated-product lane — `documentation` is in the changeset `ignore` list but **does** release with normal semver via the "Release apps & documentation" PR, so it takes a real-semver changeset). Shipped via the repo's PR conventions (`shipping-a-pull-request`) after all gates pass.

## Risks and accepted changes

- **Next/PostCSS `@plugin`(.ts)/`@source` support is unverified in-repo** — the top risk, de-risked by the Phase 0 spike before any fan-out. Fallback identified (precompiled plugin entry / CSS shim).
- **Dark theme is net-new design** in the shared package. Mitigated by mirroring the light theme's construction and using docs' current dark palette as the quality bar, verified visually. Low blast radius (no app exercises default dark today).
- **Font change is visible** (Quicksand → Inclusive Sans/Nunito) — intended under full adoption.
- **Static export + fonts/CSP**: docs is served via Netlify headers; self-hosted `data:`/`self` fonts must be allowed. Verified in the built `out/` output, not just dev (per the `@layer`/lightningcss-flattening caution).
- **Prose styling** (`@tailwindcss/typography`) must be re-bound to fresco tokens so long-form docs content reads correctly in both themes — covered in the component pass, verified in the visual sweep.

## Out of scope

Restructuring docs content or the MDX authoring model; adding new locales/i18n content; replacing the Algolia search or the bespoke MDX renderer with anything new (they are re-themed, not rewritten); any architect/interviewer change beyond confirming the shared dark theme doesn't regress them; upstreaming docs-specific components into fresco-ui.
